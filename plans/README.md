## Dependency notes

None of these are *logical* dependencies (no plan's correctness relies on
another having landed first) — they're **same-file conflict avoidance**
recommendations for whoever is dispatching executors:

- 002, 007, and 008 all edit `package.json`'s `dependencies` block. Run them
  in numeric order (002 → 007 → 008) or rebase between them to avoid a stale
  diff clobbering a prior removal.
- 002 and 011 both edit `CLAUDE.md`. Run 002 first (it rewrites the "Content
  & MDX" pipeline description) so 011's edit (the layout-divergence bullet)
  applies against the already-updated file.
- 009 and 010 both edit `app/blog/[slug]/page.tsx` and
  `app/project/[slug]/page.tsx`. Run 009 first (it extracts the shared
  `MDXRemote` options) so 010's one-line variant fix applies cleanly after.

If plans are dispatched to isolated worktrees in parallel instead, these
conflicts will surface as ordinary merge conflicts on rebase — resolve by
re-applying the smaller diff (007/008/010/011) on top.

## Findings considered and rejected

- **`eslint-plugin-tailwindcss` v3 vs. Tailwind v4 config resolution** — real,
  already self-documented in `eslint.config.mjs:50-52`, but the claimed stderr
  noise wasn't independently reproduced during vetting (truncated in the
  verification lint run). Not planned; re-audit if it becomes a visible
  annoyance.
- **No CI / pre-commit hooks, loose `^` dependency ranges, stale
  `bun.lockb` ESLint ignore entry, boilerplate README** — all real but low
  urgency for a solo-maintained personal site with no team to protect from
  silent breakage. Bundle into one pass later if the site gains contributors
  or a CI budget becomes worthwhile.
- **Adding a unit/characterization test suite** — for this repo (static
  content, no auth, no data mutation), `next build` already fails hard on
  TypeScript errors and on malformed MDX frontmatter (Zod `.parse()` throws
  in `lib/mdx.ts`). That's an adequate verification baseline as-is; a
  conventional test suite would mostly cover presentational components with
  no branching logic. Revisit if `lib/mdx.ts`'s filtering logic grows real
  branching (e.g. multi-tag AND/OR filters, pagination).
- **`components/code-group.tsx` (`CodeGroup`)** — exported from
  `mdx-components.tsx` but not currently used by any content file. Left as an
  available primitive rather than flagged as dead code, since (unlike the
  rehype-pretty-code cluster in plan 002) nothing about it is unreachable —
  it's just unused today.
